# Архитектурный обзор — Исполнительное резюме
## ANPR System v0.8 Web

**Дата обзора:** 2026-04-14  
**Рецензент:** Автоматизированный углублённый анализ (Claude Sonnet 4.6)  
**Ветка:** dev_db_ref

---

## Общая оценка

Проект имеет **хорошую основу**: понятную многослойную архитектуру, корректное внедрение зависимостей, центральный `AppContainer`, параметризованный SQL во всех местах, абстракцию OCR на основе протоколов и достаточно хорошо протестированный доменный слой. Качество архитектуры выше среднего для активного MVP.

Однако есть **несколько конкретных проблем**, которые по мере роста проекта приведут к сложностям в сопровождении, рискам безопасности или скрытым ошибкам во время выполнения. Это не архитектурные катастрофы, но большинство из них стоит устранить до первого продакшен-релиза.

## Основные риски (по приоритету)

| # | Риск | Серьёзность | Уверенность |
|---|------|-------------|-------------|
| 1 | XSS: значения из базы данных вставляются через `innerHTML` в 5 JS-файлах | high | high |
| 2 | `refresh_storage_clients()` заменяет `container.lists_db`, но работающий `ChannelProcessor` продолжает держать старую ссылку — проверки по спискам в цикле распознавания тихо используют устаревший объект | high | high |
| 3 | `_io_pool` (`ThreadPoolExecutor`) в `ChannelProcessor` никогда не останавливается — потоки записи файлов могут переживать завершение процесса | medium | high |
| 4 | `channel_runtime.py:549` напрямую читает `pipeline.aggregator._track_states` — жёсткая привязка к приватному состоянию, всё сломается при изменении внутренней реализации `TrackAggregator` | medium | high |
| 5 | Блоки миграционной защиты `DO $$` в `schema.sql` выполняются при каждом холодном старте на чистой БД — это лишняя работа и одновременно скрытие реального вида схемы | medium | high |
| 6 | `ListDatabase._schema_sql()` содержит 7 выражений `ALTER TABLE … ADD COLUMN IF NOT EXISTS`, которые нужны только для обратной совместимости со старыми установками | medium | high |
| 7 | Значение по умолчанию `JWT_SECRET_KEY = "anpr-default-secret-change-me"` в `auth_utils.py` — слишком легко угадать, если переменная окружения не задана в деплое | medium | high |
| 8 | Индексы `idx_events_timestamp` и `idx_events_channel_id` уже покрываются более широкими составными индексами в схеме — это лишний расход места и накладные расходы на запись | low | high |
| 9 | `ClientDatabase._schema_sql()` возвращает `SELECT 1` — скрытая неявная зависимость: `ListDatabase._ensure_schema()` должен выполниться первым на том же DSN | medium | medium |
| 10 | Daemon-поток создаётся прямо внутри HTTP-обработчика (`update_channel`) для `sync_channel_runtime` — fire-and-forget без нормальной поверхности ошибок | low | high |

## Возможности для очистки с наивысшим приоритетом

1. **Исправить XSS** — заменить использование `innerHTML` с пользовательскими данными на `textContent` / `createElement` в `events.js`, `clients.js`, `lists.js`, `journal.js`, `debug.js`. Атакующий может внедрить HTML через госномер или имя клиента, сохранённые в БД.

2. **Синхронизировать `processor._lists_db` после `refresh_storage_clients()`** — исправление в одну строку: после пересоздания `self.lists_db` вызвать `self.processor._lists_db = self.lists_db`.

3. **Удалить 7 строк `ALTER TABLE … ADD COLUMN IF NOT EXISTS` из `ListDatabase._schema_sql()`** — `CREATE TABLE` уже определяет все столбцы. Эти строки нужны только для старых установок, где этих столбцов ещё не было. При чистой БД это мёртвый код, который запускается на каждом старте.

4. **Удалить миграционные блоки `DO $$` из `schema.sql`** — оба защищённых блока `ALTER TABLE` (`plate_display`, `password_changed_at`) уже отражены в определении `CREATE TABLE`. Их нужно убрать.

5. **Добавить `_io_pool.shutdown(wait=False)` в teardown `ChannelProcessor`** — это предотвратит ситуацию, когда потоки записи файлов остаются жить после остановки процесса.

6. **Добавить `TrackAggregator.has_active_tracks()`** — нужно предоставить публичный метод, чтобы `channel_runtime.py` не читал `aggregator._track_states` напрямую.

## Оценка кода миграций / backfill / compatibility

**Такой код существует — и его можно агрессивно удалять:**

| Расположение | Тип | Безопасно удалить? |
|--------------|-----|--------------------|
| `schema.sql` строки 17-26, 49-58 | Защитные миграционные блоки (`DO $$`) | Да — столбцы уже есть в `CREATE TABLE` |
| `lists_repository.py` `_schema_sql()` строки 62-74 | Backfill-столбцы через `ALTER TABLE` | Да — `CREATE TABLE` уже определяет их |
| `settings_normalizer.py:76-79` | Удаление старого поля `export_dir` | Да — это было нужно для старого формата конфига |
| `settings_normalizer.py:121-123` | Удаление старого `ocr.confidence_threshold` | Да — это было нужно для старого формата конфига |
| `settings_migrations/runner.py` `_apply_legacy_compat()` | Legacy-путь для конфигов (без `settings_lineage`) | Условно — нужен только если ещё существуют YAML-конфиги без ключа lineage |

Сам `run_settings_migrations` должен остаться; кандидатом на удаление является только ветка legacy-compat внутри него, и то лишь после того, как все уже развернутые конфиги будут обновлены.

## Итоговая оценка

| Область | Оценка | Комментарий |
|---------|--------|-------------|
| Разделение архитектуры | 8/10 | Чистые слои, хорошее внедрение зависимостей |
| Согласованность нейминга | 6/10 | Алиас `e` для clients, лишние префиксы `plate`, местами вводящие в заблуждение названия |
| Безопасность | 5/10 | XSS в 5 JS-файлах, дефолтный JWT secret |
| Чистота схемы БД | 6/10 | Остатки миграционной логики в двух местах |
| Риски производительности | 7/10 | Есть мелкие проблемы, критических узких мест не найдено |
| Поддерживаемость | 7/10 | Доступ к приватному состоянию, fire-and-forget потоки, неявный порядок инициализации схемы |

**Итог: 6.5/10 — хорошая база, а список конкретных задач по очистке вполне управляемый.**
