# Аутентификация и управление пользователями

## Общее

Все API-эндпоинты, кроме `/api/health` и `POST /api/auth/login`, требуют валидный JWT-токен. Токен получается через вход с логином и паролем.

---

## JWT-токены

- Получается через `POST /api/auth/login`.
- Передаётся в заголовке `Authorization: Bearer <token>` или query-параметром `?token=<jwt>` (используется для SSE и MJPEG-стримов).
- Срок жизни по умолчанию: 8 часов (настраивается через `JWT_EXPIRATION_MINUTES`).

---

## Роли и модель доступа

В системе используются роли `superadmin`, `admin`, `operator`, но авторизация API построена по **permissions/capabilities** (permission-first).

- `superadmin` — технический root-аккаунт и bypass любых permission-проверок.
- `admin` / `operator` — получают только явно назначенные ключи.
- `operator` не может получать `tab:settings` и `users:manage`.

Разрешения хранятся как JSONB-массив строковых ключей в записи пользователя.

### Tab-права (видимость UI)

| Ключ | Название | Описание |
|---|---|---|
| `tab:obs` | Наблюдение | Доступ к вкладке live-наблюдения |
| `tab:journal` | Журнал | Доступ к журналу событий |
| `tab:clients` | Клиенты | Доступ к вкладке клиентов и списков номеров |
| `tab:settings` | Настройки | Доступ к вкладке настроек |

### Capability-права (операции API)

| Ключ | Назначение |
|---|---|
| `users:manage` | Управление пользователями |
| `settings:read` | Чтение глобальных настроек и списка стран |
| `settings:write` | Изменение глобальных настроек |
| `controllers:manage` | Управление аппаратными контроллерами |
| `data:manage` | Retention, экспорт, backup/restore |
| `debug:read` | Просмотр debug state и debug logs |
| `debug:write` | Изменение debug-настроек |

## Сводка по уровням доступа

| Что запрашивается | Требование |
|---|---|
| Health check, вход | Публичный доступ — токен не нужен |
| Все остальные эндпоинты | Валидный JWT (любой активный пользователь) |
| Управление пользователями | `users:manage` |
| Глобальные настройки | `settings:read` / `settings:write` |
| Контроллеры | `controllers:manage` |
| Данные и backup/restore | `data:manage` |
| Debug чтение | `debug:read` |
| Debug изменение | `debug:write` |

## Аккаунт по умолчанию

При первом запуске, если таблица пользователей пуста, автоматически создаётся аккаунт:

| Поле | Значение |
|---|---|
| Логин | `superadmin` |
| Пароль | `1234` |
| Роль | `superadmin` |

**Перед использованием в production немедленно смените пароль.**

Поле `warn_default_password: true` в ответе на вход сигнализирует, что пароль ни разу не менялся.

## Web UI — вход и выход

- При открытии страницы, если в `localStorage` нет валидного токена (`anpr_token`), отображается экран входа (`#login-overlay`).
- После успешного входа JWT сохраняется в `localStorage` под ключом `anpr_token`; данные пользователя хранятся в `state.currentUser`.
- В правом углу topbar отображается логин текущего пользователя и кнопка «Выход».
- Выход очищает `anpr_token` из `localStorage` и перезагружает страницу.
- Все API-запросы из UI (включая SSE и MJPEG-стримы) передают JWT через `Authorization: Bearer` или `?token=`.

## Auth API endpoints

| Метод | URL | Доступ | Описание |
|---|---|---|---|
| `POST` | `/api/auth/login` | Публичный | Аутентификация по логину + паролю; возвращает JWT и данные пользователя |
| `POST` | `/api/auth/logout` | Авторизованный | Фиксирует выход в аудит-лог; клиент удаляет токен |
| `GET` | `/api/auth/me` | Авторизованный | Возвращает текущего пользователя (роль, разрешения) |
| `GET` | `/api/permissions/available` | `tab:settings` | Возвращает список доступных ключей разрешений с метаданными |

**Защита от брутфорса:** `POST /api/auth/login` допускает не более 5 неудачных попыток в минуту с одного IP. Счётчик сбрасывается при успешном входе; при превышении возвращается `HTTP 429`.

## Управление пользователями

Все эндпоинты управления пользователями требуют разрешение `tab:settings` (или роль `superadmin`).

| Метод | URL | Описание |
|---|---|---|
| `GET` | `/api/users` | Список пользователей (аккаунт `superadmin` не отображается) |
| `POST` | `/api/users` | Создать пользователя (только роли `admin` или `operator`) |
| `GET` | `/api/users/{id}` | Получить пользователя по ID |
| `PUT` | `/api/users/{id}` | Обновить роль, разрешения или статус активности |
| `PUT` | `/api/users/{id}/password` | Сменить пароль (admin меняет любому; пользователь — только свой) |
| `DELETE` | `/api/users/{id}` | Деактивировать пользователя (мягкое удаление, `is_active = false`) |

Пользователи никогда не удаляются физически.

**Защита от самоблокировки:**
- Нельзя деактивировать собственную учётную запись.
- Нельзя снять роль `superadmin` с себя, если вы единственный активный superadmin.

---

## Web UI — управление пользователями

Доступно в разделе **Настройки → Пользователи** (отображается пользователям с разрешением `tab:settings`).

- Список: логин, роль (АДМ / ОПЕ), статус активности.
- Создание: логин, пароль, роль, разрешения по вкладкам.
- Редактирование: роль, разрешения, переключатель активен/неактивен, смена пароля.

| Переменная | По умолчанию | Описание |
|---|---|---|
| `JWT_SECRET_KEY` | `anpr-default-secret-change-me` | Секрет подписи JWT. **Обязательно заменить перед production-деплоем.** |
| `JWT_EXPIRATION_MINUTES` | `480` | Срок жизни токена в минутах (по умолчанию 8 часов) |
