# Парковочные зоны

Режим зон позволяет отслеживать въезд и выезд транспортных средств по парковочным зонам. Каждая зона имеет вместимость, а система в реальном времени показывает количество занятых и свободных мест.

---

## Концепция

**Зона** — именованная парковочная область с заданной вместимостью. К зоне привязываются каналы ANPR-камер: одни отвечают за въезд, другие за выезд. Система отслеживает факт въезда и выезда каждого ТС по его номерному знаку.

**Канал въезда (`entry`)** — при распознавании номера создаёт событие с полями `zone_id` и `time_entry`.

**Канал выезда (`exit`)** — при распознавании номера ищет последнее открытое событие въезда для этого номера в данной зоне и записывает в него `time_exit`. Новое событие не создаётся.

---

## Настройка

### 1. Создать зону

Перейти на вкладку **Зоны** и нажать **Создать зону**. Задать:
- **Название** — произвольное (например, «Парковка А»)
- **Вместимость** — максимальное количество мест (0 = без ограничения)

### 2. Привязать каналы к зоне

В настройках каждого канала (вкладка **Каналы → Конфигурация**) задать:
- **Зона** — выбрать из списка
- **Тип канала** — `Въезд` или `Выезд`

Один канал может принадлежать только одной зоне. Одна зона может иметь несколько въездных и несколько выездных каналов.

### 3. Режим фильтрации и зоновый допуск

Поле `list_filter_mode` канала управляет и реле контроллера, и записью зоновых полей:

| Режим | Зоновые поля записываются для... |
|---|---|
| `all` | Любого ТС, кроме внесённых в чёрный список |
| `whitelist` | ТС из списков типа `white`; чёрный список блокирует |
| `custom` | ТС из выбранных списков; чёрный список блокирует |

Реле контроллера срабатывает по тем же правилам независимо от записи зоновых полей.

---

## Поведение при событиях

### Въезд

1. Камера въезда распознаёт номер.
2. Проверяется зоновый допуск (`list_filter_mode`).
3. Если допуск есть — создаётся событие с `zone_id` и `time_entry = <текущее время>`.
4. Если допуска нет — событие создаётся, но без зоновых полей.
5. Реле срабатывает по результату той же проверки.

### Выезд

1. Камера выезда распознаёт номер.
2. Проверяется зоновый допуск.
3. Если допуск есть — находится последнее открытое событие въезда для этого номера в этой зоне (где `time_exit IS NULL`), и в него записывается `time_exit`. Поле `zone_id` этого события устанавливается в `0` (sentinel «выехал»). Новая запись в БД не создаётся.
4. Если открытое событие въезда не найдено — ничего не записывается, реле всё равно срабатывает.
5. Реле срабатывает по результату той же проверки.

## Отображение занятости

На вкладке **Зоны** для каждой зоны показывается:

```
Вместимость: 50   Занято: 12   Свободно: 38
```

**Занято** — количество событий в таблице `events`, где `zone_id = <id зоны>` и `time_exit IS NULL`. Вычисляется при каждом запросе, не кешируется.

Для обновления данных перейдите на вкладку повторно или обновите страницу. Автоматического real-time push в v1 нет.

---

## Поле `zone_id` в событиях

| Значение | Смысл |
|---|---|
| `NULL` | Событие не связано с зоной (обычный канал без зоны) |
| `> 0` | ТС находится в зоне; значение — идентификатор зоны |
| `0` | ТС выехало из зоны (sentinel); `time_exit` заполнен |

Поле не является внешним ключом (FK). Исторические события сохраняются даже после удаления зоны — `zone_id` продолжает ссылаться на уже несуществующую зону как на историческую метку.

---

## Удаление зоны

При удалении зоны (кнопка **Удалить** на вкладке Зоны):

1. Frontend запрашивает `GET /api/zones/{id}` и показывает список привязанных каналов.
2. После подтверждения вызывается `DELETE /api/zones/{id}`.
3. В одной транзакции: у всех каналов с `zone_id = <id>` поля `zone_id` и `zone_channel_type` сбрасываются в `NULL`, затем зона удаляется.
4. Существующие события с этим `zone_id` **не изменяются** — они остаются как исторические записи.

После удаления список каналов обновляется автоматически.

---

## Известные ограничения

**Несколько открытых въездов одного номера.** Если одно ТС дважды проехало въезд без промежуточного выезда, в зоне будет два открытых события. Выезд закроет только самое последнее из них (по `time DESC`). Более раннее останется открытым и будет учтено в подсчёте занятости. Это ожидаемое поведение в v1.

**Въезд до активации фичи.** Если ТС въехало до настройки зон, открытого события въезда не будет. Выезд не найдёт записи для обновления, реле сработает, но `time_exit` не запишется.

**Нет cross-zone валидации.** Система не отслеживает, находится ли ТС одновременно в нескольких зонах. Каждая зона ведёт учёт независимо.

**Нет real-time push занятости.** Счётчик занятых мест обновляется только при переходе на вкладку Зоны или обновлении страницы.

## API

| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/zones` | Список зон с `id`, `name`, `capacity`, `occupied`, `free` |
| `POST` | `/api/zones` | Создать зону `{name, capacity}` |
| `GET` | `/api/zones/{id}` | Детали зоны + список привязанных каналов |
| `PUT` | `/api/zones/{id}` | Обновить `name` и/или `capacity` |
| `DELETE` | `/api/zones/{id}` | Удалить зону, сбросить назначения каналов |

Поля зоны в ответе канала (`GET /api/channels`, `GET /api/channels/{id}/config`):

| Поле | Тип | Описание |
|---|---|---|
| `zone_id` | `integer \| null` | Идентификатор зоны; `null` — зона не назначена |
| `zone_channel_type` | `"entry" \| "exit" \| null` | Тип канала в зоне |

Поля зоны в событии (`GET /api/events`, SSE):

| Поле | Тип | Описание |
|---|---|---|
| `zone_id` | `integer \| null` | `null` — без зоны; `0` — выехал; `> 0` — в зоне |
| `time_entry` | `string (ISO 8601) \| null` | Момент въезда |
| `time_exit` | `string (ISO 8601) \| null` | Момент выезда |
