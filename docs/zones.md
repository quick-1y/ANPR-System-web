# Парковочные зоны

Режим зон позволяет отслеживать въезд и выезд транспортных средств по парковочным зонам. Каждая зона имеет вместимость, а система в реальном времени показывает количество занятых и свободных мест.

---

## Концепция

**Зона** — именованная парковочная область с заданной вместимостью. К зоне привязываются каналы ANPR-камер: одни отвечают за въезд, другие за выезд. Система отслеживает факт въезда и выезда каждого ТС по его номерному знаку.

**«Вне парковки» (`zone_id = 0`)** — специальное системное понятие, означающее, что ТС находится снаружи парковочной зоны. Это зарезервированное значение, не хранящееся в таблице зон и не требующее создания. Оно всегда доступно как опция при настройке канала под названием **«Вне парковки»**. Вместимость «Вне парковки» не ограничена.

**Канал въезда (`entry`)** — при распознавании номера создаёт событие с `zone_id = zone_after`, `time_entry`, `channel_id_entry`, `frame_path_entry` и `plate_path_entry`.

**Канал выезда (`exit`)** — при распознавании номера ищет последнее открытое событие въезда для этого номера в зоне `zone_before` и записывает в него `time_exit`, `zone_id = zone_after`, `channel_id_exit`, `frame_path_exit` и `plate_path_exit`. Найденная запись обновляется и поднимается вверх ленты/журнала по времени выезда. Если открытого въезда нет, создаётся отдельное событие выездной попытки без канала/кадров въезда, а реле не вызывается.

## Настройка канала

### Поля зоны в настройках канала

Каждый канал настраивается тремя полями:

| Поле | Смысл |
|---|---|
| **Зона до** (`zone_before_id`) | Зона, в которой находится ТС **до** прохождения канала |
| **Зона после** (`zone_after_id`) | Зона, в которой оказывается ТС **после** прохождения канала |
| **Тип канала** (`zone_channel_type`) | `Въезд` (entry) или `Выезд` (exit) |

Поле «Тип канала» становится активным только тогда, когда заданы обе зоны («Зона до» и «Зона после»).

### Типичная конфигурация

| Канал | Зона до | Зона после | Тип |
|---|---|---|---|
| Въезд в подземную парковку | Вне парковки | Подземная парковка | Въезд |
| Выезд из подземной парковки | Подземная парковка | Вне парковки | Выезд |

### Пошаговая настройка

**1. Создать зону**

Перейти на вкладку **Зоны** и нажать **Создать зону**. Задать:
- **Название** — произвольное (например, «Подземная парковка»)
- **Вместимость** — максимальное количество мест (0 = без ограничения)

**2. Привязать каналы**

В настройках каждого канала (вкладка **Каналы → Конфигурация**) задать:
- **Зона до** — зона или «Вне парковки»
- **Зона после** — зона или «Вне парковки»
- **Тип канала** — `Въезд` или `Выезд`

Один канал принадлежит одной паре зон. Одна зона может иметь несколько въездных и несколько выездных каналов.

**3. Режим фильтрации и зоновый допуск**

Поле `list_filter_mode` канала управляет и реле контроллера, и записью зоновых полей:

| Режим | Зоновые поля записываются для... |
|---|---|
| `all` | Любого ТС, кроме внесённых в чёрный список |
| `whitelist` | ТС из списков типа `white`; чёрный список блокирует |
| `custom` | ТС из выбранных списков; чёрный список блокирует |

Реле контроллера срабатывает по тем же правилам независимо от записи зоновых полей.

## Поведение при событиях

### Логика веток A / B / C

Внутренняя обработка события в runtime делится на три ветки:

- **Ветка A** — канал без активной зоновой конфигурации (`zone_before_id` или `zone_after_id` не заданы, либо `zone_channel_type = NULL`).
- **Ветка B** — канал настроен как **въезд** (`zone_channel_type = "entry"`).
- **Ветка C** — канал настроен как **выезд** (`zone_channel_type = "exit"`).

Дальнейшие подразделы ниже описывают поведение веток B и C в деталях.

### Въезд

1. Камера въезда распознаёт номер.
2. Проверяется зоновый допуск (`list_filter_mode`).
3. Если допуск есть — создаётся событие с `zone_id = zone_after_id`, `time_entry = <текущее время>`, `channel_id_entry`, `frame_path_entry` и `plate_path_entry`.
4. Если допуска нет — событие создаётся, но без зоновых полей.
5. Реле срабатывает по результату той же проверки.

### Выезд

1. Камера выезда распознаёт номер.
2. Проверяется зоновый допуск.
3. Если допуск есть — находится последнее открытое событие для этого номера, где `zone_id = zone_before_id` и `time_exit IS NULL`. В найденное событие записываются `time_exit`, `zone_id = zone_after_id`, `channel_id_exit`, `frame_path_exit` и `plate_path_exit`; поле `time` обновляется временем выезда, чтобы запись была видна оператору вверху ленты и журнала.
4. Если открытое событие не найдено — создаётся отдельное событие выездной попытки с `time_exit`, `channel_id_exit`, `frame_path_exit`, `plate_path_exit` и пустыми полями въезда. Реле в этом сценарии не вызывается.
5. Если номер не проходит допуск по спискам или находится в чёрном списке, событие выездной попытки также создаётся без вызова реле.

> Поле `zone_id` в событии всегда отражает **текущую зону** ТС после прохождения канала. Это `zone_after_id` для обоих типов — въезда и выезда.

## Отображение занятости

На вкладке **Зоны** для каждой зоны показывается:

```
Вместимость: 50   Занято: 12   Свободно: 38
```

**Занято** — количество событий в таблице `events`, где `zone_id = <id зоны>` и `time_exit IS NULL`. Вычисляется при каждом запросе, не кешируется.

Для обновления данных перейдите на вкладку повторно или обновите страницу. Автоматического real-time push в v1 нет.

---

## Поле `zone_id` в событиях

| Значение | Смысл |
|---|---|
| `NULL` | Событие не связано с зоной (канал без зоновой конфигурации) |
| `0` | «Вне парковки» — ТС находится снаружи зоны (например, после выезда) |
| `> 0` | ТС находится в указанной зоне; значение — идентификатор зоны |

Поле не является внешним ключом (FK). Исторические события сохраняются даже после удаления зоны — `zone_id` продолжает ссылаться на уже несуществующую зону как на историческую метку.

«Вне парковки» (`zone_id = 0`) не является строкой в таблице `zones`. Это зарезервированный системный sentinel. В списке управления зонами он не отображается.

---

## Удаление зоны

При удалении зоны (кнопка **Удалить** на вкладке Зоны):

1. Frontend запрашивает `GET /api/zones/{id}` и показывает список привязанных каналов.
2. После подтверждения вызывается `DELETE /api/zones/{id}`.
3. В одной транзакции: у всех каналов, которые ссылаются на эту зону в `zone_before_id` или `zone_after_id`, поля `zone_before_id`, `zone_after_id` и `zone_channel_type` сбрасываются в `NULL`, затем зона удаляется.
4. Существующие события с этим `zone_id` **не изменяются** — они остаются как исторические записи.

После удаления список каналов обновляется автоматически.

---

## Известные ограничения

**Несколько открытых въездов одного номера.** Если одно ТС дважды проехало въезд без промежуточного выезда, в зоне будет два открытых события. Выезд закроет только самое последнее из них (по `time DESC`). Более раннее останется открытым и будет учтено в подсчёте занятости. Это ожидаемое поведение в v1.

**Въезд до активации фичи.** Если ТС въехало до настройки зон, открытого события въезда не будет. Выезд создаст отдельную запись выездной попытки с пустыми `time_entry` и `channel_id_entry`; реле не сработает, потому что система не подтвердила активный въезд.

**Нет cross-zone валидации.** Система не отслеживает, находится ли ТС одновременно в нескольких зонах. Каждая зона ведёт учёт независимо.

**Нет real-time push занятости.** Счётчик занятых мест обновляется только при переходе на вкладку Зоны или обновлении страницы.

## API

### Зоны

| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/zones` | Список зон с `id`, `name`, `capacity`, `occupied`, `free` |
| `POST` | `/api/zones` | Создать зону `{name, capacity}` |
| `GET` | `/api/zones/{id}` | Детали зоны + список привязанных каналов |
| `PUT` | `/api/zones/{id}` | Обновить `name` и/или `capacity` |
| `DELETE` | `/api/zones/{id}` | Удалить зону, сбросить назначения каналов |

### Зоновые поля канала

Поля, возвращаемые в ответах `GET /api/channels` и `GET /api/channels/{id}/config`:

| Поле | Тип | Описание |
|---|---|---|
| `zone_before_id` | `integer \| null` | Зона до прохождения канала; `null` — не задано; `0` — Вне парковки |
| `zone_after_id` | `integer \| null` | Зона после прохождения канала; `null` — не задано; `0` — Вне парковки |
| `zone_channel_type` | `"entry" \| "exit" \| null` | Тип канала в зоне; `null` — зоновая обработка не активна |

### Зоновые поля события

| Поле | Тип | Описание |
|---|---|---|
| `zone_id` | `integer \| null` | `null` — без зоны; `0` — Вне парковки; `> 0` — в зоне |
| `time_entry` | `string (ISO 8601) \| null` | Момент въезда |
| `time_exit` | `string (ISO 8601) \| null` | Момент выезда |
| `channel_id_entry` | `integer \| null` | Канал, на котором зафиксирован въезд |
| `channel_id_exit` | `integer \| null` | Канал, на котором зафиксирован выезд |
| `frame_path_entry` / `plate_path_entry` | `string \| null` | Кадр распознавания и кадр номера для въезда |
| `frame_path_exit` / `plate_path_exit` | `string \| null` | Кадр распознавания и кадр номера для выезда |
