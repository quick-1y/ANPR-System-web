# Architecture

Этот документ описывает архитектуру ANPR System на уровне сервисов, потоков данных и ключевых runtime-компонентов. Mermaid-схемы вынесены в [`diagrams.md`](diagrams.md), а этот файл служит текстовым объяснением того, как система устроена и как её части взаимодействуют между собой.

## High-Level Overview

Система состоит из четырёх основных частей:

- `nginx` — единая публичная точка входа
- `api` — основной FastAPI-сервис с Web UI, REST API и runtime обработки каналов
- `retention_worker` — фоновый сервис для очистки данных, retention и export-задач
- `postgres` — основное хранилище событий, пользователей, клиентов, списков, каналов и контроллеров

Оператор работает через браузер. Web UI обращается к API по HTTP, получает live-события через SSE и preview-потоки через MJPEG.

## Main Runtime Flow

Для каждого активного канала создаётся отдельный поток обработки. Типичный путь данных выглядит так:

1. `ChannelProcessor` открывает источник видео через `cv2.VideoCapture(...)`.
2. Кадры поступают в детекцию и ANPR pipeline.
3. Pipeline формирует итоговый номер, направление и метаданные события.
4. По распознанному номеру выполняется поиск клиента через `ListDatabase.find_client_by_plate()`. Клиент ищется среди всех активных записей в таблице `clients` — независимо от того, прикреплён ли он к списку. Если совпадение найдено, `client_id` записывается в событие; иначе — `NULL`.
5. Если у канала задана зоновая конфигурация (`zone_before_id`, `zone_after_id` и `zone_channel_type`):
   - **Канал въезда (`entry`)** — событие записывается в БД с `zone_id = zone_after_id` и `time_entry` (при наличии зонового допуска).
   - **Канал выезда (`exit`)** — вместо создания нового события находится последнее открытое событие для этого номера, где `zone_id = zone_before_id`; в него записывается `time_exit` и `zone_id = zone_after_id`. Новая запись не создаётся.
   - Если зоновая конфигурация не задана — поведение не меняется.
6. Событие публикуется в `EventBus` и сразу появляется в Web UI через SSE.
7. Preview-кадры отдаются клиенту как snapshot или MJPEG-поток.

> **Фильтр по спискам и `client_id` — независимые механизмы.** Настройка канала `list_filter_mode` (`all` / `whitelist` / `custom`) управляет исключительно тем, сработает ли реле контроллера (шлагбаум/турникет). На поиск клиента и запись `client_id` в событие этот режим не влияет: обогащение события происходит всегда, до любой проверки фильтра.

> **`zone_id` в событии — текущая зона ТС.** Всегда отражает `zone_after_id` канала, через который прошло ТС: для въезда — зона, в которую оно въехало; для выезда — зона, в которую оно вышло (типично `0` = «Вне парковки»). Значение `0` — системный sentinel «Вне парковки», не хранящийся в таблице `zones`. Значения `> 0` ссылаются на конкретную зону (без FK). `zone_id IS NULL` — событие не связано с зоной.

Таким образом, один и тот же runtime одновременно обслуживает распознавание, историю событий и live-наблюдение без отдельного медиасервера.

## Service Responsibilities

### `nginx`

`nginx` выступает внешним reverse proxy. Он принимает HTTP-трафик, проксирует запросы к API и обеспечивает единую публичную точку входа для Web UI и операторских запросов.

### `api`

Основной FastAPI-сервис отвечает за:

- Web UI
- REST API
- JWT-аутентификацию и проверки прав
- запуск и управление каналами
- preview endpoints
- SSE-потоки live-событий и live-логов
- работу с PostgreSQL через репозитории и сервисы приложения

Внутри этого сервиса находится и channel runtime, и операторский API, поэтому именно `api` является центральной точкой всей системы.

### `retention_worker`

Отдельный worker отвечает за сервисные фоновые операции:

- очистку старых событий
- удаление устаревших медиафайлов
- контроль лимитов хранилища
- export и связанные задачи обслуживания данных

Это разгружает основной API-сервис и выносит циклические операции в отдельный процесс.

### `postgres`

PostgreSQL — основное хранилище системы. В базе находятся:

- события распознавания
- зоны парковки
- пользователи
- клиенты
- списки номеров
- каналы
- контроллеры

Глобальные runtime-настройки при этом хранятся в `config/settings.yaml`, а не в PostgreSQL.

## Application Layers

### Web UI

Frontend представляет собой SPA в `app/web/`. Он отвечает за авторизацию пользователя, отображение вкладок, журнал событий, live-наблюдение, настройки, работу с клиентами и backup/restore из интерфейса.

### API Layer

Слой `app/api/routers/` разбит по предметным областям: auth, users, channels, events, clients, lists, settings, data, system и debug. Права доступа проверяются через зависимости FastAPI.

### Runtime Layer

`runtime/channel_runtime.py` управляет жизненным циклом каналов: запуском, остановкой, preview-кешем, сбором метрик и передачей кадров в ANPR pipeline.

### ANPR Core

Ядро в каталоге `anpr/` отвечает за детекцию, OCR, агрегацию результатов, постобработку и оценку направления движения. Важный принцип этой версии — трек-уровневый OCR с ограниченным бюджетом попыток, чтобы не тратить лишний CPU на длинные и нечитаемые треки.

## Data Flow Details

### Live Preview

Для preview система не использует отдельный streaming server. Кадры кодируются в JPEG и сохраняются в памяти как последние доступные preview-данные канала. Затем они выдаются через snapshot endpoint или MJPEG stream.

### Live Events

После формирования события оно публикуется в in-memory `EventBus`. Клиенты, подписанные на SSE endpoint, получают обновления почти сразу после записи события в базу.

### Storage Lifecycle

Retention-политика применяется отдельно от основного пути распознавания. Worker удаляет старые записи и orphan-media в соответствии с настройками хранения.

## Access Model

С точки зрения архитектуры прав доступа есть четыре практических уровня:

- public — health check и login
- authenticated — любой активный пользователь с валидным JWT
- requires `tab:settings` — операции управления пользователями
- superadmin only — контроллеры, глобальные настройки, backup/restore, debug и чувствительные системные операции

Эта модель должна последовательно отражаться в документации и в API-описании.
