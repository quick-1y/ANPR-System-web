# Аудит настроек (`settings`) и границы с настройками каналов

Дата аудита: 2026-05-26.

## Краткий вывод

Сейчас в проекте уже есть разделение:
- **глобальные настройки** в `config/settings.yaml`;
- **канальные настройки** в БД (`database/channel_repository.py`).

Но есть несколько признаков «размазанности» и дублей, которые стоит привести к более строгой модели.

## Что выглядит лишним или избыточным в `settings`

### 1) `logging.retention_days` — кандидат №1 на удаление

- Параметр хранится в схеме и UI.
- Фактическая очистка логов/медиа/событий выполняется через `storage` + `DataLifecyclePolicy`.
- По коду нет места, где `logging.retention_days` реально участвует в политике удаления логов.

Рекомендация:
- либо удалить `logging.retention_days` (и UI-поле);
- либо реализовать реальный lifecycle для логов на его основе.

### 2) `time.offset_minutes` — потенциально лишний (зависит от продукта)

- Это дополнительная ручная поправка поверх timezone.
- По коду есть getter в `SettingsManager`, но практическое системное применение ограничено.
- Такие ручные оффсеты часто ведут к «плавающему» времени при сопровождении.

Рекомендация:
- если нет подтверждённого сценария эксплуатации, убрать из UI/API и оставить только `timezone`;
- если нужен, документировать точный pipeline его применения.

### 3) `logging.allowed_levels` — технически вычисляемое поле

- Фактически это константный список из `LOG_LEVELS`.
- В normalizer всё равно принудительно синхронизируется с дефолтом.

Рекомендация:
- не хранить в `settings.yaml`, а отдавать из бэкенда как read-only справочник (или хардкод в UI).

## Что, наоборот, убирать не стоит

- `storage.events_retention_days`, `storage.media_retention_days`, `storage.max_screenshots_mb`, `storage.cleanup_interval_minutes`, `storage.auto_cleanup_enabled` — используются retention worker’ом и являются рабочими.
- `reconnect.*`, `models.*`, `ocr.*`, `detector.*`, `inference.*` — реально участвуют в runtime/инициализации.

## Профессиональная целевая модель (рекомендуемая)

1. **System Settings (YAML):**
   - только системные/инфраструктурные и UI-глобальные параметры.
2. **Channel Settings (DB):**
   - всё, что влияет на конкретный канал и должно жить рядом с записью канала.
3. **Контракт:**
   - запрет дублирования одного и того же параметра одновременно в YAML и channel JSON;
   - у каждого поля один source of truth.

## Приоритетный план приведения

1. Удалить или реализовать `logging.retention_days` (предпочтительно удалить как мёртвый параметр).
2. Решить судьбу `time.offset_minutes` (оставить только при подтверждённой эксплуатации).
3. Убрать persisted `logging.allowed_levels` из файла настроек (сделать вычисляемым).
4. Зафиксировать карту «какие поля системные, какие канальные» в `docs/technical/modules.md` и `README.md`.

## Риски при чистке

- Потребуется bump `settings_version` + migration path в `config/settings_migrations/`.
- Нужно синхронно обновить UI (`app/web/js/settings.js`, `app/web/index.html`) и API-схемы.
